# 全新卡牌游戏重启开发计划（从零开始）

## 1. 目标与边界

- 目标：开发一款可在 Web 端运行的单机卡牌游戏，先实现核心玩法闭环，再逐步扩展内容与系统。
- 平台：移动端优先（竖屏），兼容桌面浏览器。
- 版本策略：MVP（可玩） -> Alpha（可测） -> Beta（可扩） -> 1.0（可发布）。
- 边界：先不做联网对战、先不做账号系统、先不做复杂剧情编辑器。

## 2. 核心玩法定义（MVP）

- 战斗结构：回合制卡牌战斗（玩家 vs AI）。
- 牌组结构：固定初始牌组（20~30 张），包含攻击、防御、功能牌。
- 回合循环：
  1. 抽牌
  2. 消耗能量打牌
  3. 结算效果
  4. 敌方行动
  5. 回合结束与状态清理
- 资源系统：生命值、能量、护甲、状态（中毒/易伤等）。
- 胜负条件：敌方生命归零获胜；玩家生命归零失败。

## 3. 系统架构（重建建议）

- `core/`：纯逻辑层，不依赖 DOM。
- `content/`：卡牌、敌人、状态、掉落规则等配置数据。
- `ui/`：Web 展示层（渲染、交互、动画）。
- `app/`：启动入口与场景编排。

建议采用“数据驱动 + 纯函数结算”：
- 卡牌通过 JSON 配置描述基础属性与效果关键字。
- 战斗结算在 `core` 内完成，UI 只订阅状态变更。

## 4. 里程碑与交付物

### M0（1~2 天）：项目重置与骨架
- 建立全新目录结构与基础工程脚手架。
- 确定代码规范与命名规范。
- 交付：可运行空场景页面、基础日志系统。

### M1（3~5 天）：单场战斗 MVP
- 完成抽牌/打牌/回合切换/敌方行动。
- 实现至少 12 张基础卡（攻/防/功能）。
- 实现 2~3 个敌人模板。
- 交付：可完整打一局并判定胜负。

### M2（3~5 天）：玩法强化
- 增加状态系统（中毒、脆弱、力量等）。
- 增加基础 AI 策略（按条件选行动）。
- 增加战斗日志与可视化反馈。
- 交付：可进行多种策略尝试，具备可测性。

### M3（4~7 天）：局外成长（轻量）
- 战后奖励三选一。
- 简易牌组构筑（加牌/删牌）。
- 关卡节点（3~5 战斗节点 + 1 精英节点）。
- 交付：单局流程 15~30 分钟。

### M4（3~5 天）：发布前打磨
- 新手引导与核心规则提示。
- 平衡性首轮调优。
- 性能与移动端交互优化。
- 交付：Beta 包与发布检查清单。

## 5. 开发顺序（执行清单）

1. 先实现战斗状态机（不做 UI）。
2. 用命令行或测试脚本验证结算正确性。
3. 再接入最小 UI（手牌、敌人、结束回合按钮）。
4. 增加卡牌效果关键字与状态系统。
5. 最后补足内容量与体验优化。

## 6. 测试策略

- 单元测试：
  - 抽牌与洗牌
  - 能量消耗
  - 伤害与护甲结算
  - 状态叠加与回合衰减
- 集成测试：
  - 一整场战斗流程（可复现 seed）
  - 关键卡牌组合的结算顺序
- 回归测试：
  - 每次新增卡牌时跑固定对局集合。

## 7. 设计复盘建议（你现在可以直接做）

1. 从“核心乐趣”倒推机制：每回合决策点是什么。
2. 明确卡牌关键词上限：MVP 控制在 6~8 个关键词。
3. 定义首批内容规模：
   - 卡牌 20~30
   - 敌人 8~12
   - 状态 6~10
4. 设定一条 30 分钟完整体验路径。

## 8. 风险与规避

- 风险：机制先天不够好玩。
  - 规避：M1 结束即进行 5 轮内部试玩，先改机制再堆内容。
- 风险：卡牌效果过早复杂化。
  - 规避：先做“少量关键词 + 可组合”而不是大量特例。
- 风险：UI 先行拖慢进度。
  - 规避：逻辑先完成，UI 后置并保持可替换。

## 9. 下一步（建议从今天开始）

1. 冻结玩法定义：确认回合流程、资源类型、首批关键词。
2. 建立全新代码骨架：`core/content/ui/app`。
3. 实现第一版战斗循环与 12 张基础卡。
4. 完成首轮可玩验证后，再扩展内容与美术表现。
