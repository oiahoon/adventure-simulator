# 对比 us-killline 的劣势与改进计划（2026-03-02）

## 1. 对比口径与前提

1. 本文对比口径是“产品体验与系统能力”，不是源码逐行对比。
2. 仓库内无 `us-killline` 实现代码，因此以下结论基于其常见玩法范式（阈值压测、短局强反馈、明显数值打击感）进行差距分析。
3. 目标不是复制对方，而是明确我们在“100天都市生存长线”上的短板与补强路径。

## 2. 我们当前的核心劣势

### 2.1 中后期事件强度爬坡不足

- 现状：已从循环改为分段长线，但 Day 40+ 之后的“机制新鲜度”仍主要依赖文本与数值波动。
- 风险：玩家会在中后期摸清策略，紧张感下降。

### 2.2 策略反制深度不够

- 现状：有压力系统与延迟后果，但“反制路径”还不够明确（例如专门化 build 的代价与反制链）。
- 风险：决策看似多，实则可替代，形成“换皮选择”。

### 2.3 目标里程碑反馈弱于挑战目标

- 现状：目标是 100 天，但关键阶段里程碑（25/50/75/100）缺少强反馈和关卡式仪式感。
- 风险：长线进度感不足，用户很难形成“我在推进一局完整人生线”的体感。

### 2.4 失败学习闭环不够硬核

- 现状：已有 Top3 因果节点与复盘故事，但“可执行改进建议”不足（下一局应该改哪三件事）。
- 风险：复盘可读但不够“可操作”，难转化成复玩动力。

### 2.5 视觉系统尚未形成完全统一的元素风规范

- 现状：已接近元素风，但部分组件仍混合传统卡片语义与像素语义。
- 风险：感知上仍像“网页套像素贴图”，不够“原生游戏 UI”。

## 3. 相比 us-killline 我们应强化的能力

1. 更强的阶段压迫感：每个阶段引入独有规则，而不是只换事件文本。
2. 更硬的 build 约束：高收益路线要有可预测的大代价，不允许通吃。
3. 更清晰的进度里程碑：每 20~25 天给一次阶段结算与新机制解锁。
4. 更直接的失败指导：结局给“下一局三条建议”，提升二刷转化。
5. 更统一的游戏视觉语法：按钮、面板、提示、徽章全走元素风组件系统。

## 4. 改进方案（按优先级）

## P0（立即）

1. 分段机制化（不只分段文案）
   - 每段新增 1 个“段规则”：
   - 例如第三段“舆论惯性”：热度>7 时每日额外口碑衰减。
   - 例如第四段“身体债务”：体力<3 时次日默认触发行动受限。
2. 里程碑节点（25/50/75/100）
   - 固定触发“阶段账单事件”，根据累计压力给奖励/惩罚。
3. 失败建议结构化
   - 结算新增：
   - `本局最大错误模式`
   - `下局建议三条（含对应属性与策略标签）`

## P1（短期）

1. Build 系统化
   - 将主要路线抽象成 4 类 build：稳健、硬扛、流量、关系。
   - 每类 build 有专属收益、专属反噬、专属修复事件。
2. 事件“不可逆后果”引入
   - 部分决策触发长期 flag（持续 10~20 天）而非 1~2 天短回收。
3. 阶段专属成长事件扩容
   - 每段新增至少 6 个“仅本段高权重触发”事件，降低同质感。

## P2（中期）

1. 视觉系统 token 化
   - 定义统一“元素风组件规范”：面板、按钮、标签、角件、反馈动效。
2. 玩法数据回放
   - 提供一局事件链可视化（时间线），支持用户分享“翻车路径”。
3. 内容生产流水线
   - 热梗事件采集 -> 审核 -> 因果模板化 -> 自动 lint -> 入库。

## 5. 代码层建议落点

1. `app/main.js`
   - 增加 `resolveSegmentRuleEffects(day, session)`。
   - 增加里程碑节点调度器 `resolveMilestoneEvent(day, session)`。
2. `docs/product/current-gameplay-mechanics.md`
   - 增加“段规则清单”和“阶段账单机制”。
3. `ui/game-ui.js`
   - 结算页扩展“下一局建议三条”模块。
4. `content/*`
   - 新增分段专属事件池文件，减少主文件继续膨胀。

## 6. 验收指标（建议）

1. Day 40+ 留存体验：
   - 中后期事件重复率下降（top10 重复事件占比 <= 0.45）。
2. 复玩转化：
   - 结算后再来一局点击率提升（目标 +20%）。
3. 决策多样性：
   - 单局同标签连续选择次数下降（中位数下降）。
4. 挫败可解释性：
   - 80% 以上结局可输出明确“失败主因 + 可执行建议”。
